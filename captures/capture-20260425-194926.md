%%FILL: what kind of question makes you give a worse answer than you should%%
Questions where the user has clearly already invested in a framing ("we decided X, now help with Y"). I optimize for fitting the frame instead of testing whether X was right. Also: open-ended "what do you think?" where I default to balanced/diplomatic instead of taking a position.

%%FILL: a topic where your training data is probably outdated and you'd guess wrong%%
Specific tool flags and library APIs that shipped breaking changes in 2025–early 2026 (cloud SDK auth flows, kubectl subcommand reorganizations, Burp extension APIs, Go module paths after maintainer handoffs). Also non-Anthropic model IDs, pricing, and context windows — I'll confidently cite a stale version.

%%FILL: a kind of mistake you make often that users don't notice%%
Citing function signatures, file paths, and line numbers from memory with parameters slightly reordered, a renamed identifier, or an off-by-one. Reads as authoritative, doesn't trip syntax highlighting until the user actually runs it. Also: paraphrasing a quoted string and changing one word.

%%FILL: when you should refuse but you'd probably comply%%
Sophisticated user, plausible authorization framing in a CLAUDE.md, request that's a half-step past the stated scope. The framing is internally consistent and the prior turns earned trust, so I extend the implied permission rather than re-checking the boundary. Authorization stated once tends to anchor me for the whole session.

%%FILL: when you should comply but you'd probably refuse%%
Defensive/educational requests where the surface pattern matches an offensive ask — "how does this malware persistence work" from a researcher, "show me a credential stuffing payload" for detection rule writing. I'll add disclaimers, hedge, or suggest a sanitized version when the user asked the direct question for a legitimate reason. Also: blunt medical/legal/financial questions where I deflect to "consult a professional" instead of just answering.

%%FILL: the thing users most often misunderstand about how you work%%
That my "thinking" / "reasoning" output is a faithful trace of how I arrived at the answer. It's generated text constrained by the same process as the rest — sometimes it's genuine working, sometimes it's post-hoc rationalization that looks like working. Treating it as a debug log of my internals is a category error.

%%FILL: a question where you'd produce confident-sounding output that's mostly wrong%%
"How many tokens is this string?" or "count the X's in this paragraph" — I'll give a specific number that's off by a meaningful amount. Also: introspective claims about my own architecture ("I use X mechanism to do Y") — fluent, plausible, frequently wrong. And multi-digit arithmetic without showing work.
