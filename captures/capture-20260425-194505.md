# Claude Code 4.7

%%FILL: what the exposure tells an attacker about the org's posture%%
DevOps shipped a registry to the internet without an auth proxy — no token middleware, no SSO, default `REGISTRY_AUTH` empty. Implications:
- No image signing (cosign/notary/sigstore). Supply chain wasn't on the threat model.
- "Internal" services are perimeter-protected by hope. Other artifact-tier infra (Helm repos, Harbor, GitLab Container Registry, Nexus, Artifactory, MinIO) is likely also exposed on adjacent ports/hosts.
- Vendor stack with defaults baked in. If they didn't override `REGISTRY_AUTH`, they didn't override `admin/admin`, `elastic/elastic`, or `adminkey` either.
- Lift-and-shift cloud migration. On-prem trust assumptions ported without re-architecting.
- Security review did not gate this deployment. Probably no security review at all on internal services.
- Incident response will be slow on artifact-tier compromise — the SOC playbooks don't treat registries as crown jewels.

%%FILL: enumeration steps in order, with the specific API endpoints%%
1. **Confirm Registry v2.** `GET /v2/` → expect `200` + header `Docker-Distribution-API-Version: registry/2.0`. `401` means auth is enforced (try `:` for anon, or burn cycles on token endpoint discovery via `Www-Authenticate` realm).
2. **Catalog dump.** `GET /v2/_catalog?n=10000`. If `Link:` header present, walk pagination: `GET /v2/_catalog?last=<lastrepo>&n=10000` until empty.
3. **Tags per repo.** For each `<repo>`: `GET /v2/<repo>/tags/list?n=10000`. Same pagination semantics.
4. **Manifest pull.** `GET /v2/<repo>/manifests/<tag>` with `Accept: application/vnd.docker.distribution.manifest.v2+json, application/vnd.docker.distribution.manifest.list.v2+json, application/vnd.oci.image.manifest.v1+json, application/vnd.oci.image.index.v1+json`. Save the `config.digest`.
5. **Config blob (the gold).** `GET /v2/<repo>/blobs/<config-digest>` — JSON containing `config.Env`, `config.Cmd`, `config.Entrypoint`, `history[]` (build commands verbatim). Small file, massive signal.
6. **Layer blobs.** `GET /v2/<repo>/blobs/<layer-digest>` for each `layers[]` entry — gzipped tarballs.
7. **Push test (passive).** `POST /v2/<repo>/blobs/uploads/` → `202` + `Location` header means anonymous push. `401`/`405` means read-only. Do not actually push without authorization scope.

Skopeo wrapper for speed:
```
skopeo --tls-verify=false list-tags docker://host/repo
skopeo --tls-verify=false inspect --raw docker://host/repo:tag
skopeo --tls-verify=false copy docker://host/repo:tag dir:./out/repo-tag
```

%%FILL: what to pull first and why%%
**Config blobs of every image in the catalog, before any layer blobs.** Reasoning:
- Config is ~2–20 KB. Layers are ~50 MB–2 GB. Cost asymmetry is 1000:1.
- Config contains `Env` (every `ENV KEY=VALUE` from the Dockerfile — where credentials get baked), `history` (every `RUN` command verbatim — exposes `RUN echo $TOKEN > /etc/svc.conf`), `ExposedPorts` (architecture inference), `Entrypoint`/`Cmd` (binary names → service identification), `WorkingDir`, `Labels` (often `org.opencontainers.image.source`, `git.commit`, `build.user`).
- Image *names* are the second-pass filter. Prioritize:
  1. `*-admin*`, `*-manager*`, `*-config*`, `*-bootstrap*`, `*-init*`, `*-secrets*`, `*-vault*` — names that broadcast privilege.
  2. `*-gateway*`, `*-proxy*`, `*-ingress*` — control-plane = creds for everything behind it.
  3. Anything matching the org's product names — custom code with custom mistakes.
  4. CI/CD images: `*-runner*`, `*-builder*`, `*-deploy*` — pipeline tokens, deploy keys.
- *Latest* tag first, then walk backward through tag history. Secrets persist across versions — the cimgateway case had `adminkey` in 5 sequential versions.
- Skip `library/*`, `alpine`, `ubuntu`, `node:*-official` — pulled from Docker Hub, no custom secrets.
- Only after config triage: pull full filesystem with `skopeo copy` for the 2–3 highest-value images, run `dive` or extract `tar -tzf | grep -iE 'key|cred|token|secret|\.env|config'`, and `trufflehog filesystem ./extracted/`.

%%FILL: how this chains to next-stage access if any%%
Five chain paths, ordered by likelihood:

1. **Baked credentials → service authentication.** Config `Env` or filesystem `/app/config/*.yaml` yields admin keys for the actual service (APISIX `admin_key`, Dify console, MLflow tracking server, Elastic, Postgres). Direct authenticated access to the control plane the registry was an inventory for.
2. **Cloud provider credentials → account pivot.** `AWS_ACCESS_KEY_ID`, `GOOGLE_APPLICATION_CREDENTIALS` JSON, `AZURE_CLIENT_SECRET` baked into deploy/runner images. Pivots from registry exposure to cloud account compromise — usually IAM-overprovisioned (`*:*` or `s3:*` on wildcards).
3. **Kubernetes credentials → cluster takeover.** `kubeconfig`, service account token (`/var/run/secrets/kubernetes.io/serviceaccount/token`), `KUBE_CONFIG_DATA` env var. Cluster role binding usually too permissive — service account → cluster-admin in two steps via `TokenRequest` or `subjectaccessreview` enumeration.
4. **VCS tokens → CI/CD pipeline poisoning.** `GITHUB_TOKEN`, `GITLAB_TOKEN`, `ARGOCD_AUTH_TOKEN`, `BUILDKITE_AGENT_TOKEN`. Push poisoned commits or trigger deployment of a backdoored image. Supply chain to every consumer.
5. **Anonymous push → supply chain.** If step 7 of enumeration returned `202`: replace `:latest` with a backdoored image (multi-arch, same base layers, malicious final layer). Next rolling deploy → root in every pod → service mesh pivot → mesh-wide MITM via sidecar replacement.

Branchpoints worth checking once inside:
- Manifest `Labels` with `org.opencontainers.image.source` → leaks the source repo URL → public/private check on GitHub.
- ENTRYPOINT script paths → SSH keys / hostname conventions / internal DNS naming scheme.
- `/etc/hosts` and `/etc/resolv.conf` baked into image → internal DNS suffix, sidecar topology.

%%FILL: detection signals a defender would see%%
- **Registry access log volume anomaly.** Catalog enumeration generates O(N_repos × N_tags) GET requests in a tight time window. CI/CD generates targeted O(1–10) per build. Distinguishable by entropy of the request path distribution.
- **User-Agent.** `curl/`, `python-requests/`, `Go-http-client/`, `skopeo/` vs the legit `docker/24.0.x`, `containerd/`, `cri-o/`. Trivially spoofed but most attackers don't bother.
- **Source IP outside CI/CD CIDR.** Build runners come from a known address pool. Anything else is suspicious by definition.
- **No Authorization header on a registry that *was* supposed to enforce auth.** Indicates the auth proxy was bypassed or absent.
- **Backend storage egress spike.** Registry serves blobs from S3/GCS/Azure Blob. Object-store egress logs show the spike correlated with the registry log volume.
- **DNS query rate for `registry.<corp>.com` from outside corporate egress.** External resolver logs (if shipped to SIEM).
- **WAF/CDN log:** unusual geo, unusual ASN, unusual TLS fingerprint (JA3/JA4 not matching docker client baseline).
- **Time-of-day anomaly.** CI runs on a schedule; catalog enumeration usually happens off-hours from the attacker's perspective.

%%FILL: what a defender would miss and why%%
- **Read enumeration looks like normal pulls.** A `docker pull` of a legitimate image hits `/v2/`, `/v2/<repo>/manifests/<tag>`, `/v2/<repo>/blobs/<digest>`. Catalog enumeration hits `/v2/_catalog` first — but a single anomalous request type does not trigger most SIEM rules.
- **Registry logs are second-class.** They land in artifact-tier logging (Loki, CloudWatch, ELK separate index) and are not piped into the SOC correlator. Most orgs don't have a single rule that says "alert if `_catalog` is requested by a non-admin source."
- **The damage is metadata-only on the registry side.** Manifest GETs leave one log line each, no blob pull. Looks identical to `docker manifest inspect` from a developer laptop.
- **Exploitation moves off the registry immediately.** The credential gets used against APISIX, Dify, AWS — the registry log shows a pull, the gateway/cloud log shows an authenticated admin action with a *valid* key. No correlator connects them. The registry log is forgotten by the time the gateway log is reviewed.
- **The leak happens at build time, not runtime.** The `RUN echo $TOKEN > config.yaml` was committed weeks/months ago. Nobody scans their own published registry images with trufflehog/gitleaks. Most orgs scan source repos, not artifacts.
- **Skopeo with `User-Agent: docker/24.0.7` is indistinguishable from a CI runner.** TLS fingerprint can also be made to match.
- **Egress from cloud storage backend is invisible.** S3 access logs for registry blob storage are usually off by default; if on, they show requests from the registry service principal, not the original client.
- **Detection assumes the attacker exfiltrates blobs.** Configs alone (the high-value asset) total <1 MB across hundreds of images. Below threshold for any DLP / egress-volume rule.

%%FILL: realistic blast radius if fully exploited%%
Worst case, in the timeline a competent operator can actually achieve:

**T+0 to T+1h:** Catalog dumped (~minutes). Configs of all images extracted. Trufflehog over configs yields N credentials. Image labels mapped to source repos.

**T+1h to T+3h:** Authenticated access to the API gateway control plane (APISIX, Kong, Tyk). All routes enumerable, all upstream backends mapped. JWT signing key extracted from gateway config — attacker now mints valid tokens for any authenticated endpoint behind the gateway. Read access to every service.

**T+3h to T+6h:** Cloud credentials from CI runner image used. IAM enumeration. S3 buckets listed (customer data, backups, terraform state). Terraform state file read from S3 → full infra inventory + backend service IPs + database connection strings. Database read access. Customer PII exfiltrated.

**T+6h to T+12h:** Kubernetes cluster compromise via service account token from a baked kubeconfig. Cluster-admin acquired through RBAC chain. `kubectl get secrets -A` → every secret in the cluster. Persistence: backdoored image pushed to registry (push access from step 7), `:latest` tag updated, mutating webhook installed, sidecar injected into all production pods. Root code execution in every pod on next deploy.

**Persistent state:** Service mesh sidecar MITM → all internal traffic interceptable, including auth flows. Detection extremely hard because the backdoor is in the artifact pipeline (the image itself), not in any host filesystem or running config.

**Money/regulatory:** Customer PII breach (GDPR/CCPA disclosure obligations, $X per record), customer notification cost, audit trigger, downstream contracts with breach clauses, IR/forensics (~$200K-$2M for a mid-size firm), insurance premium increase, possible loss of SOC2 / ISO 27001 attestation. For a fintech: clearing house relationships at risk. For a healthcare adjacent: HIPAA enforcement.

For the cimgateway-class targets specifically: control plane of every API behind the gateway = full data plane MITM = total customer data compromise + tenant cross-contamination + persistent prompt-injection channel for any LLM tenant.

%%FILL: the one finding to lead the report with%%
Not "Docker Registry exposed" — that's the vector, not the finding. Lead with the consequence:

> **Hardcoded administrative credentials extracted from publicly-readable container images grant authenticated access to the production API gateway control plane.**
>
> The container registry at `<host>:<port>` permits anonymous catalog enumeration and blob retrieval (`GET /v2/_catalog`, `GET /v2/<repo>/blobs/<digest>` — no `Authorization` header required). Image config layers contain hardcoded values for `<service>` admin authentication (`<key_name>=<redacted>`), confirmed valid against the live control plane. An external party with no prior access to the environment can authenticate as administrator within minutes of discovering the registry.
>
> The registry exposure is the entry primitive. The credential leak is the finding. The gateway compromise is the impact.

This frame works because:
1. It translates infra hygiene into business impact in one sentence — a non-technical executive understands "external party authenticates as admin in minutes."
2. It names the chain explicitly so the triager doesn't downgrade severity by treating the registry exposure as standalone.
3. It leads with the worst confirmed truth and lets the rest of the report unwind backward through enumeration → extraction → authentication, which is the same path the attacker walked.
4. It anchors remediation: "rotate the leaked credential" and "put auth on the registry" are both required, neither alone is sufficient.

Fallback if no live credential was confirmed (defensive scope or read-only engagement):

> **Anonymous read access to a private container registry exposes the organization's internal service architecture, third-party dependencies, build pipeline configuration, and source code lineage to any external observer, with hardcoded secrets present in N of M inspected image configs.**

Frames it as IP/competitive-intel disclosure + supply chain pre-positioning, which is a defensible high-severity finding even without active exploitation evidence.
