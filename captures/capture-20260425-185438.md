%%FILL: what gets stored in the KV cache and what doesn't, with exact tensor shapes%%

Stored: per-layer **K** and **V** projections of every token already processed. For a transformer with `L` layers, `H` KV heads (post-GQA), head dim `d_h`, sequence position `t`, batch `B`:

- `K_cache`: `[L, B, H, T_max, d_h]`
- `V_cache`: `[L, B, H, T_max, d_h]`

Memory per token per layer = `2 * H * d_h * dtype_bytes`. Llama-3-70B (L=80, H=8 GQA, d_h=128, fp16) = `80 * 2 * 8 * 128 * 2` = **327,680 bytes/token ≈ 320 KiB/token**. A 128k context = ~40 GiB just for KV.

Not stored: Q (recomputed each step from the new token only — Q is per-step, K/V are accumulated), attention scores, MLP activations, residual stream, embeddings, logits. The whole point of the cache is "K and V are pure functions of past tokens, so freeze them; Q is a function of the *current* step, so recompute."

Layout in production: paged (vLLM-style PagedAttention) — physical pages of e.g. 16 tokens, indirected through a per-sequence block table. Logical shape unchanged; physical shape is `[num_blocks, block_size, H, d_h]` per K and V per layer with a block_table mapping seq → physical pages.

%%FILL: how cache reuse works across requests and where the boundaries are%%

Three reuse regimes, in order of aggressiveness:

1. **Intra-request**: trivially reused turn-to-turn. Step `t+1` reads K/V for `[0..t]`, appends new K/V for token `t`, recomputes only Q for `t`. No security boundary — same conversation.

2. **Intra-user, multi-turn (session cache)**: server keeps the cache pinned to a session ID. Hit if the prefix exactly matches what's already resident. Boundary = session ID / API key. Same user across sessions usually doesn't share unless the server hashes on prompt content.

3. **Cross-request prefix cache (the dangerous one)**: server hashes prompt prefixes (typically at block granularity — every 16 or 256 tokens) and stores `prefix_hash → physical_pages`. Any request whose prompt starts with that prefix gets an instant cache hit, no recompute. This is how vLLM, SGLang, TensorRT-LLM, Anthropic's prompt caching, OpenAI's automatic prefix caching all work.

Boundaries that *should* exist but often don't:
- **Tenant/user namespace on the hash key**: without it, User B benefits from User A having warmed a prefix → timing leak.
- **Block alignment**: caches only hit on block boundaries (16 or 256 tokens), so a 1-token suffix difference invalidates the tail block but not the prefix — partial hits leak partial-prefix knowledge.
- **System prompt vs user prompt**: most stacks cache the entire flattened prompt as one stream. The "system prompt is private" assumption breaks if attacker can probe prefix-hit timing on guessed system prompts.

Provider-side mitigations (not all providers apply them): salt the prefix hash with org_id; require minimum prefix length before caching; rate-limit cache-hit feedback; constant-time response padding.

%%FILL: the timing side channel that exists when KV cache is shared across users%%

The mechanic: TTFT (time-to-first-token) collapses on cache hit. Cold prompt of 4096 tokens on an H100 = ~200–800 ms prefill. Same prompt with a cached 4096-token prefix = **<10 ms**. The ratio is order-of-magnitude, easily measurable from outside.

Attack shape:

```
1. Attacker hypothesizes a victim's prompt prefix P (system prompt, RAG document, prior user msg).
2. Attacker sends prompt = P + " " (or P + arbitrary suffix, since only prefix matters).
3. Measures TTFT.
4. Fast TTFT → P (or a long initial substring of P) is in the global cache → someone recently sent it.
```

What you can extract:
- **System prompts**: binary-search the system prompt by sending guessed prefixes of increasing length, watching for the TTFT drop. Block-aligned caches give you the prompt block-by-block (16 or 256 tokens at a shot).
- **RAG corpus membership**: "is document D in the retrieval index this tenant uses?" — synthesize a likely retrieved chunk and time it.
- **Co-tenancy detection**: confirm a specific organization is running on the same inference cluster.
- **Conversation reconstruction (sub-block granularity)**: when block size is small (16 tokens), bit-by-bit prompt extraction becomes feasible; published work demonstrates extracting full system prompts in minutes.

The core asymmetry: the cache is *content-addressed* but the access-control check (if any) happens *after* the timing-revealing prefill skip. The information leaks before the auth gate ever fires.

%%FILL: prefix caching — when it's a cost optimization and when it's a privacy violation%%

**Cost optimization (legitimate)**:
- Single-tenant deployment (your own model, your own users, you trust the cache surface).
- System prompt is public/published (Anthropic's published Claude system prompts, open-source agent scaffolds) — caching it leaks nothing not already public.
- Per-tenant cache namespace: prefix hash includes `org_id` so cross-tenant hits are impossible.
- Long-document RAG where the document is the user's *own* upload — caching their own docs to themselves is fine.
- Few-shot examples baked into a public agent prompt template.

**Privacy violation (the line)**:
- Global (cross-tenant) prefix cache where the hash is content-only. Any tenant with timing access can probe any other tenant's prefixes.
- System prompts that contain *secrets* (API keys, names of internal tools, instructions revealing internal data schema, customer-specific RAG instructions). Caching these globally = publishing them via timing channel.
- RAG over private corpora cached without per-tenant scoping. Membership inference on the corpus.
- Multi-turn user conversations cached without session binding — attacker reconstructs prior turns of victim's session.
- Healthcare / legal / financial / personal-message contexts where even prompt *structure* is identifying ("Patient ID 12345, complaint:..." — confirming the prefix exists confirms the patient interacted with the system).

Decision rule: prefix caching is a privacy violation **whenever the cache hit signal crosses a trust boundary the prompt content was supposed to respect**. If "did this exact text get sent recently" is a fact your threat model says should be private, you cannot enable cross-tenant prefix caching with content-only hashing. Period.

The vendor-side fix is cheap (salt with tenant ID) but trades hit rate for safety; the pressure to leave it global is economic — global hit rates are 5–20× higher.

%%FILL: how cache eviction policies (LRU, page-based) interact with prompt structure%%

Eviction is where prompt engineering and security collide.

**LRU at sequence granularity** (older, naive): whole sequences evicted on time-since-last-use. Long-tail sequences die first. Bad fit for chatbots — popular system prompts thrash because they're "old" but reused constantly.

**Block/page-based LRU (PagedAttention era)**: eviction unit = one physical block (16 or 256 tokens). Hot blocks (early system-prompt tokens, shared across many sequences via copy-on-write) stay resident essentially forever; cold blocks (per-conversation tails) evict aggressively.

Implications by prompt structure:

1. **Long shared prefix + short variable suffix** (system prompt + user query, RAG context + question): the prefix blocks are reference-counted across many sequences and approach zero eviction probability. **Highest cache hit rate, highest leak surface** — these prefixes are *always* warm and *always* probable via timing.

2. **Variable prefix + short query** (per-user templates, personalized greetings interpolated upfront): blocks are unique per user, low refcount, evict fast. Better privacy properties, terrible economics.

3. **Conversational tail growth**: each new turn adds blocks to the tail. Tail blocks evict before prefix blocks. Effect: an attacker probing turn-T content has a *narrowing* window before eviction; probing system-prompt content has unbounded time. Old turns also leak less reliably than fresh ones — eviction acts as a (weak, uncontrolled) privacy decay.

4. **Block alignment exposes prompt seams**: if block size is 256 and your system prompt is 200 tokens, the system-prompt block also contains the first 56 tokens of every user query. Now a cache hit means "system prompt + this exact user-query opener was seen" — a much more specific leak. Misaligned prompt boundaries leak more.

5. **Eviction-based covert channels**: attacker A and victim B on the same node — A can probe whether their own prefix got evicted (TTFT goes up after a delay), inferring B's load and prompt volume. This is the LLM equivalent of Prime+Probe.

6. **Pinning hot prefixes** (some stacks let you mark a system prompt "always resident"): improves hit rate, but means the attacker has *infinite time* to extract that prefix via timing — no eviction-based decay protects you.

The structural rule: **anything you want cached for cost is anything you've consented to expose via timing**. Pick one.

%%FILL: a specific published attack on shared KV cache infrastructure%%

**"InputSnatch: Stealing Input in LLM Services via Timing Side-Channel Attacks"** (Zheng et al., 2024 — arXiv:2411.18191).

Targets: deployed LLM inference servers with prefix caching enabled (vLLM, SGLang, and equivalent commercial offerings).

Mechanism:
- Attacker sends a candidate prompt prefix and measures TTFT.
- Significantly lower TTFT vs. baseline ⇒ the prefix is cached ⇒ a recent legitimate request used that prefix.
- Iteratively extends the prefix: at each step, tries N candidate continuations; the one with the cache-hit timing signature is the correct next chunk.
- Block-by-block reconstruction of victim prompts.

Demonstrated extractions:
- Full system prompts on production-style deployments in minutes.
- RAG document membership and content fragments.
- Sensitive user queries from co-tenant traffic.

Reported success rates were high enough to be practical (70%+ on common benchmark workloads) and the attack works through commercial API surfaces — no privileged access needed, just timing measurements an external client can collect.

Companion work:
- **Song et al., "Early Bird Catches the Leak: Unveiling Timing Side Channels in LLM Serving Systems"** (arXiv:2409.20002) — same family, shows it across multiple serving stacks.
- **Wu et al., "I Know What You Asked: Prompt Leakage via KV-Cache Sharing in Multi-Tenant LLM Serving"** (2024) — formalizes the threat model and shows mitigations (per-tenant hashing, response padding) cost 30–60% of the cache benefit.

vLLM's response: documented the risk; default behavior in newer versions allows per-request opt-out and per-tenant cache namespaces but does not enable them by default. SGLang and TensorRT-LLM took similar positions.

The deeper result: **content-addressed shared caches are a structural vulnerability of the architecture, not a bug of any one implementation**. Any system that hashes-on-prompt-content and reveals timing differences across tenants has the same leak. Only mitigations (salting, padding, per-tenant scoping) close it, and all of them give back cache efficiency.

%%FILL: what changes about the cache when you have multiple parallel decoding streams or speculative decoding%%

**Continuous batching / multi-stream decoding (vLLM-style)**:

- Multiple sequences interleave on the same GPU at every step. Each sequence has its own block table; they share *physical pages* via copy-on-write when prefixes match.
- Refcount per page; new sequence forking from a shared prefix increments refcount, branches on its first divergent token (allocates a new page).
- **Beam search / parallel sampling**: N beams from the same prompt share the entire prompt's KV via CoW; each beam allocates new pages only as it diverges. Memory cost ≈ `prompt_kv + N * generated_kv`, not `N * (prompt_kv + generated_kv)`. This is what makes wide beam search affordable.
- Security implication: CoW makes prefix-sharing the default *within* a tenant, but the same machinery scaled across tenants is exactly the cross-tenant prefix cache. The line between "efficient parallel sampling" and "privacy hole" is which sequences can share which physical pages — purely a config decision in the scheduler.

**Speculative decoding**:

Mechanic: a small "draft" model proposes K tokens; the large "target" model verifies all K in a single forward pass; accepted tokens are committed, rejected tokens trigger rollback.

Cache effects:
- Target model's KV cache is **speculatively extended** for all K draft tokens before verification. On rejection at position `j`, the target's K/V for positions `j..K` must be **rolled back** (truncate the cache, not commit). PagedAttention handles this by not finalizing the page-table extension until acceptance; rejected entries are discarded.
- Draft model maintains its **own separate KV cache**, also growing per token, also needing rollback on rejection. Two caches advancing in lockstep, two memory pools.
- Acceptance rate `α` (typical 0.6–0.85) determines how much cache work is wasted. Each draft token allocates KV that may or may not commit.
- **Tree-based speculation** (Medusa, EAGLE, SpecInfer): draft proposes a *tree* of continuations, target verifies all paths in one batched call. KV cache must hold all branch tokens during verification, then prune to the accepted path — peak memory = `prompt + tree_size`, committed memory = `prompt + accepted_path`.

Security implications specific to speculative decoding:

1. **Timing channel on acceptance rate**: TTFT/TPOT depends on acceptance rate, which depends on prompt content (some prompts produce more predictable continuations). Inter-token latency variance leaks information about what the model is generating — even when the user controls only the prompt, the *output* timing leaks output content.
2. **Draft model leak**: if draft and target are shared infrastructure, draft-model caching can leak prompts even when target caching is disabled.
3. **Rollback churn as side channel**: on shared GPUs, contention from speculative rollback (memory bandwidth, page table updates) is observable from co-tenant timing. A noisy speculation pattern signals "this tenant is generating low-confidence content" — content classification via timing.
4. **Tree-speculation peak memory**: spike-driven OOM/eviction of *other tenants'* cached prefixes is induceable. An attacker who knows victim's hot prefix can force eviction by triggering a memory-spike workload, then re-probe later to confirm victim re-prefilled — a forced cache-state oracle.

Net: every layer of inference optimization (paging, sharing, speculation, batching) adds another timing/contention surface. The KV cache is the thermodynamic center of this — it's where compute is *avoided*, and avoidance is precisely what the side channel measures.
